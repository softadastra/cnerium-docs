# Project Structure

A Cnerium project should stay simple, readable, and easy to grow.

The recommended structure is based on a normal Vix project, with Cnerium used as the web framework layer.

## Minimal application

A minimal Cnerium application can live entirely in `src/main.cpp`.

```cpp
#include <cnerium/app/app.hpp>
#include <vix/console.hpp>

using namespace cnerium::app;

int main()
{
  App app;

  app.get("/", [](AppContext &ctx)
  {
    ctx.text("Hello from Cnerium");
  });

  app.listen("127.0.0.1", 8080, []()
  {
    vix::console.info("Cnerium app is ready");
  });
}
```

This is enough for a first app.

## Create the project

Create a new Vix project:

```bash
vix new api
cd api
```

Add Cnerium:

```bash
vix add cnerium/app
```

Run the app:

```bash
vix dev
```

## Important files

### `vix.json`

`vix.json` is the project manifest.

It stores project metadata, dependencies, and reusable tasks.

Example:

```json
{
  "name": "api",
  "version": "0.1.0",
  "type": "app",
  "dependencies": {
    "cnerium/app": "^0.5.0"
  },
  "tasks": {
    "dev": "vix dev",
    "run": "vix run",
    "build": "vix build",
    "check": "vix check --tests",
    "fmt": "vix fmt"
  }
}
```

Use `vix add cnerium/app` instead of editing dependencies manually.

### `vix.lock`

`vix.lock` stores the exact resolved dependency versions.

Commit it to Git.

It makes installs reproducible across machines.

### `.vix/`

`.vix/` is generated by Vix.

It can contain local dependencies and CMake integration files.

Typical generated files:

```txt
.vix/
в”њв”Ђв”Ђ deps/
в””в”Ђв”Ђ vix_deps.cmake
```

Do not edit this folder manually.

### `CMakeLists.txt`

`CMakeLists.txt` defines the C++ target.

A simple app usually has one executable target.

### `src/main.cpp`

`src/main.cpp` is the application entry point.

For small projects, it can contain the full app.

For larger projects, keep it focused on wiring:

```cpp
int main()
{
  App app;

  register_middleware(app);
  register_routes(app);

  app.listen("127.0.0.1", 8080);
}
```

## `handlers/`

Use `handlers/` for route logic.

```txt
include/api/handlers/
в”њв”Ђв”Ђ HealthHandlers.hpp
в””в”Ђв”Ђ UserHandlers.hpp
```

Example:

```cpp
#pragma once

#include <cnerium/app/app.hpp>

namespace api::handlers
{
  inline void register_health_routes(cnerium::app::App &app)
  {
    app.get("/health", [](cnerium::app::AppContext &ctx)
    {
      ctx.json({
        {"ok", true},
        {"status", "healthy"}
      });
    });
  }
}
```

Then in `main.cpp`:

```cpp
#include <api/handlers/HealthHandlers.hpp>

int main()
{
  cnerium::app::App app;

  api::handlers::register_health_routes(app);

  app.listen("127.0.0.1", 8080);
}
```

## `middleware/`

Use `middleware/` for reusable middleware.

```txt
include/api/middleware/
в”њв”Ђв”Ђ AppHeaders.hpp
в””в”Ђв”Ђ AuthGuard.hpp
```

Example:

```cpp
#pragma once

#include <cnerium/app/app.hpp>

namespace api::middleware
{
  inline void register_app_headers(cnerium::app::App &app)
  {
    app.use([](auto &ctx, auto next)
    {
      ctx.response().set_header("X-App", "Cnerium");
      next();
    });
  }
}
```

## `models/`

Use `models/` for domain structures.

```txt

в””в”Ђв”Ђ User.hpp
```

Example:

```cpp
#pragma once

#include <string>

{
  struct User
  {
    int id{};
    std::string name;
    std::string email;
    bool active{true};
  };
}
```

## `services/`

Use `services/` for business logic.

```txt
include/api/services/
в””в”Ђв”Ђ UserService.hpp
```

Handlers should stay thin.

They should call services instead of containing all business logic directly.

## `repositories/`

Use `repositories/` for persistence logic.

```txt
include/api/repositories/
в””в”Ђв”Ђ UserRepository.hpp
```

At the beginning, you can keep users in memory.

Later, this folder can contain database access.

## `config/`

Use `config/` for app configuration helpers.

```txt
include/api/config/
в””в”Ђв”Ђ AppConfig.hpp
```

Example:

```cpp
#pragma once

#include <cnerium/app/app.hpp>

namespace api::config
{
  inline cnerium::app::AppConfig make_config()
  {
    cnerium::app::AppConfig config;
    config.host = "127.0.0.1";
    config.port = 8080;
    config.thread_count = 4;
    return config;
  }
}
```

## Clean `main.cpp`

For a growing application, keep `main.cpp` small.

```cpp
#include <cnerium/app/app.hpp>
#include <vix/console.hpp>

#include <api/config/AppConfig.hpp>
#include <api/handlers/HealthHandlers.hpp>
#include <api/middleware/AppHeaders.hpp>

int main()
{
  cnerium::app::App app(api::config::make_config());

  api::middleware::register_app_headers(app);
  api::handlers::register_health_routes(app);

  app.listen([]()
  {
    vix::console.info("Cnerium app is ready");
  });
}
```

This keeps the entry point readable.

## Recommended growth path

Start simple:

```txt
src/main.cpp
```

Then split when needed:

```txt
src/main.cpp
include/api/handlers/
```

Then add middleware:

```txt
include/api/middleware/
```

Then add models and services:

```txt
include/api/models/
include/api/services/
```

Then add persistence:

```txt
include/api/repositories/
```

Do not create all folders before you need them.

## Naming convention

Use clear names.

Good:

```txt
User.hpp
UserService.hpp
UserRepository.hpp
HealthHandlers.hpp
AuthGuard.hpp
AppHeaders.hpp
```

Avoid vague names:

```txt
Utils.hpp
Manager.hpp
Stuff.hpp
Helpers.hpp
```

A helper file is fine when the role is precise, but the name should explain the domain.

## Route registration convention

Prefer functions that register routes on `App`.

```cpp
namespace api::handlers
{
  void register_user_routes(cnerium::app::App &app);
}
```

Then call them in `main.cpp`:

```cpp
api::handlers::register_user_routes(app);
```

This pattern scales well because each feature owns its route registration.

## Middleware registration convention

Use the same style for middleware:

```cpp
namespace api::middleware
{
  void register_app_headers(cnerium::app::App &app);
  void register_auth_guard(cnerium::app::App &app);
}
```

Then:

```cpp
api::middleware::register_app_headers(app);
api::middleware::register_auth_guard(app);
```

Order matters.

Middleware runs in the order it is registered.

## Keep route order clear

Cnerium routes are matched in insertion order.

Register specific routes before dynamic routes when needed:

```cpp
app.get("/users/me", handler);
app.get("/users/:id", handler);
```

This prevents `/users/me` from being treated as a dynamic `id`.

## Where to put JSON helpers

For small apps, JSON helpers can stay near the handler.

For larger apps, place them near the model or service:

```txt
include/api/models/User.hpp
include/api/models/UserJson.hpp
```

Example:

```cpp
#pragma once

#include <api/models/User.hpp>
#include <cnerium/json/json.hpp>

namespace api::models
{
  inline cnerium::json::value to_json(const User &user)
  {
    return cnerium::json::object{
        {"id", user.id},
        {"name", user.name},
        {"email", user.email},
        {"active", user.active},
    };
  }
}
```

## Where to use `vix::console`

Use `vix::console` for application logs.

```cpp
#include <vix/console.hpp>

vix::console.info("Cnerium app is ready");
vix::console.warn("Using development configuration");
vix::console.error("Failed to start app");
```

Use it in:

```txt
startup callbacks
background tasks
runtime tasks
error reporting
development diagnostics
```

## Where to use `vix::print`

Use `vix::print` for simple output examples, scripts, or demos.

```cpp
#include <vix/print.hpp>

vix::print("Hello from Cnerium");
```

In application code, prefer `vix::console` for logs.

## Recommended `vix.json` tasks

Use tasks to keep common workflows short.

```json
{
  "tasks": {
    "dev": "vix dev",
    "run": "vix run",
    "fmt": "vix fmt",
    "check": "vix check --tests",
    "test": "vix tests",
    "build": "vix build",
    "ci": [
      "vix fmt --check",
      "vix check --tests"
    ],
    "release": [
      "vix build --preset release",
      "vix check --preset release --tests"
    ]
  }
}
```

Then run:

```bash
vix task dev
vix task ci
vix task release
```

## Summary

A Cnerium project should start small.

Use `src/main.cpp` first.

As the app grows, split by responsibility:

```txt
handlers/
middleware/
models/
services/
repositories/
config/
```

Keep `main.cpp` focused on wiring.

Keep handlers thin.

Keep services responsible for business logic.

Use `vix::console` for application logs.

Use `vix::print` for simple output examples.
